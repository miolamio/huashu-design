# Self-Critique Rubric — 5-Axis Scoring (Pre-Delivery)

> Detailed rubric for the **Self-critique (mandatory final pass)** step in `SKILL.md` §Workflow Step 10. Score the deliverable 0-40 across 5 axes. If `<30`, regenerate **once**. If still `<30`, ship with a quality flag.
>
> The rubric is borrowed from the existing `references/critique-guide.md` (which is for critiquing someone else's work). The scoring axes match by design — whether the agent is grading its own work or the user's, the criteria are the same. **Do not invent a different rubric for self-critique** — alignment is more important than novelty.

## Why This Step Exists

The Discover-phase research (2026-04) identified "forced self-look before delivery" as the **single highest-leverage missing primitive** across every design skill in the ecosystem, including Anthropic's `frontend-design`. Models hallucinate CSS that "should work" and ship broken layouts because they never look at their own output. A 2-3 minute self-critique catches the majority of "ugh, the agent shipped broken nav" failures.

**The discipline isn't the score** — it's the *forced second look*. Most slop ships because the agent never grades its own work.

## The 5 Axes (0-10 each, 40 total)

These are the same axes used in `references/critique-guide.md`. Each axis has anchored bands.

### Axis 1 · Philosophy Alignment (0-10)

How completely does the deliverable embody the chosen design direction (POV + archetype)?

| Score | Criteria |
|---|---|
| 9-10 | Every detail has philosophical grounding; an outside reviewer could name the archetype and POV from the design alone. |
| 7-8 | Overall direction correct, core traits present, a few details drift. |
| 5-6 | Intent visible but execution mixes elements from other archetypes; lacks purity. |
| 3-4 | One or two surface gestures toward the archetype but the rest is generic. |
| 0-2 | No discernible POV/archetype; output is the AI-default average. |

### Axis 2 · Visual Hierarchy (0-10)

Does the layout direct the eye in the order the POV requires?

| Score | Criteria |
|---|---|
| 9-10 | First fixation lands on the load-bearing element; second and third fixations follow the POV's argument. |
| 7-8 | Hierarchy works overall; one or two elements compete for attention. |
| 5-6 | The most important element is present but doesn't dominate; eye must hunt. |
| 3-4 | Multiple equally-weighted elements; no clear focal point. |
| 0-2 | Flat — every element fights for the same visual space. |

### Axis 3 · Craft Quality (0-10)

How does the deliverable handle micro-details: type kerning, color values, alignment to grid, transition timing, edge cases?

| Score | Criteria |
|---|---|
| 9-10 | Pixel-perfect alignment, deliberate type pairing, brand-spec colors used consistently, motion timed in service of meaning. |
| 7-8 | Generally tight; one or two minor alignment or color drift issues. |
| 5-6 | Most details correct but visible inconsistencies (mixed border-radius, inconsistent spacing, off-by-2px alignment). |
| 3-4 | Multiple craft errors visible at first scan. |
| 0-2 | Sloppy: default spacing values, system fonts, broken layouts on standard viewports. |

### Axis 4 · Functionality (0-10)

Does the deliverable work mechanically? Does it match the technical contract for its kind (slides scale to 1920×1080, animations set `window.__ready`, app prototypes pass click tests)?

| Score | Criteria |
|---|---|
| 9-10 | Zero console errors. All interactive elements work. Verified via `verify.py`. |
| 7-8 | Works in primary path; one minor issue in an edge case. |
| 5-6 | Primary path works; multiple edge-case bugs. |
| 3-4 | Primary path partially broken; user can see the issue immediately. |
| 0-2 | Doesn't work at all on first load. |

### Axis 5 · Originality (0-10)

Does the deliverable look like AI-default, or does it have a recognizable point of view?

| Score | Criteria |
|---|---|
| 9-10 | Distinctive enough that a designer would recognize the deliberate choice; passes the "AI made this?" test verbatim. |
| 7-8 | Has personality but borrows several common patterns. |
| 5-6 | Competent but generic — would not stand out in a portfolio. |
| 3-4 | Multiple AI-slop patterns present (purple gradients, rounded cards + left border, SVG illustrations, Inter as display, etc.) |
| 0-2 | Pure AI-slop archetype — could have been generated by any default LLM. |

## Pass Threshold and Regeneration

| Total score | Action |
|---|---|
| **30-40** | Ship. Optional: note any axis ≤6 in the delivery summary so the user knows where to direct further iteration. |
| **20-29** | **Regenerate once.** Identify the lowest-scoring axis and target the fix at that axis specifically (don't rewrite the whole thing). Re-score. |
| **20-29 on second pass** | Ship with a **Quality flag** (see next section). Do not silently ship a sub-30 deliverable. |
| **0-19 on second pass** | Stop and ask the user. Likely the brief is mis-scoped or the POV is wrong. |

**Cap at one regeneration.** Uncapped self-critique loops cause one of two failure modes: (a) the agent regresses to the mean as it averages successive attempts, or (b) the agent inflates its own scores to escape the loop. Either way, infinite regeneration is the trap.

## Quality Flag Format

When shipping a `<30` deliverable on second pass, prepend the delivery summary with:

```
Quality flag: 24/40

Weak axes:
- Originality (4/10): hero section uses default Tailwind cards; no clear archetype committed
- Visual Hierarchy (5/10): pricing CTA does not dominate; competes with feature grid

Recommended next steps:
- Pick an archetype from references/named-allowlist.md (likely Editorial or Startup given POV)
- Replace feature grid with a single anchor element directing toward CTA
```

This is **honest signaling**, not self-deprecation. The user benefits from knowing where the deliverable is weak so they can target their iteration.

## Anti-Patterns

These ways of doing self-critique fail and should be avoided:

- **Inflating to escape the loop**. Scoring a clearly-mediocre output as 31/40 to skip regeneration. The cap-at-one rule exists to prevent this; if you find yourself wanting to inflate, ship with the Quality flag instead.
- **Regenerating from scratch**. The whole-thing rewrite usually averages back to a similar score. **Target the lowest-scoring axis** in the regeneration. If craft is 4/10, fix the craft issues; don't redesign the layout.
- **Self-grading without looking at the deliverable**. The score must follow `verify.py` screenshot inspection. Grading from memory of what you intended produces inflated scores.
- **Treating self-critique as performative**. The point isn't to look thorough — it's to catch real problems. If the score is 38/40, write 2 lines and ship. If it's 22/40, write the full Quality flag.
- **Using a different rubric for self-critique vs user-requested critique**. Both use this file's 5 axes. Inconsistency confuses users about what the score means.
